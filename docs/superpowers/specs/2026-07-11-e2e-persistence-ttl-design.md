# BeamChat 端到端加密、消息持久化与 TTL 设计

## 范围

本次变更通过一次协议升级完成 issue #3、#5 和 #7。BeamChat 继续保持匿名、无账号的产品定位，同时为房间消息、私聊消息、中继及点对点文件、以及中继及直连语音增加经过身份认证的端到端加密。服务端可以持久化消息密文和 TTL 设置，但不得保存消息明文、文件明文、语音明文、房间密码、身份私钥或会话私钥。

## 安全模型

每个客户端安装实例拥有一组持久 Ed25519 身份密钥，以及一个由该身份密钥签名的持久 X25519 离线预密钥。Ed25519 公钥的 SHA-256 摘要作为用户可读的设备指纹。每次连接还会生成临时 X25519 密钥，并由 Ed25519 身份密钥签名。通信双方必须先验证签名，再派生共享密钥。

对端身份采用首次信任（TOFU）。首次看到某个联系人名称对应的合法身份公钥时，将其保存在本地。后续公钥发生变化时，冻结该会话的加密发送功能，并显示旧指纹和新指纹，直到用户明确接受或拒绝。删除本地身份状态或重装客户端后，已有联系人会看到身份变化警告。

Ed25519 私钥、持久及临时 X25519 私钥、已信任的对端密钥和消息偏移量仅存放在客户端状态目录。平台支持时，私钥文件必须设置为仅文件所有者可访问。协议不得传输任何私钥。

## 密码学方案

- 身份签名：Ed25519。
- 共享密钥协商：在线会话使用临时 X25519；离线接收方密钥信封使用带签名的持久 X25519 预密钥。
- 密钥派生：HKDF-SHA256，并将协议、作用域和参与者信息加入上下文。
- 消息、文件和语音加密：ChaCha20-Poly1305。
- 密码派生：使用现有打包依赖能够提供的 Argon2id；若依赖不可用，构建必须失败，不得静默降级。
- 每次 AEAD 操作都必须通过附加认证数据绑定协议版本、作用域、发送方身份、内容类型和序列号或消息标识。
- 同一密钥下 nonce 必须唯一。持久化消息和文件密钥包装使用随机 nonce；有序文件分块和语音包使用单调递增计数器。接收端在有界重放窗口内拒绝重复计数器。

## 协议状态机

连接状态依次为 `CONNECTING -> HELLO -> IDENTIFYING -> READY -> REJOINING`。客户端发送的第一帧必须是 `CLIENT_HELLO`；在 HELLO 成功前，服务端不得发送应用层欢迎帧。HELLO 帧包含协议版本、软件版本、必需能力、身份公钥、带签名的持久 X25519 预密钥、临时 X25519 公钥及其 Ed25519 签名。协议版本不支持、缺少必需能力、编码无效或签名无效时，返回结构化且不可恢复的 `PROTOCOL_INCOMPATIBLE` 或 `CRYPTO_INVALID` 错误。

收到 `SERVER_HELLO` 后，客户端发送 `SET_NAME`。服务端仅为已完成身份标识的连接发布对端密钥包，并在身份被接受后发送 `READY`。重连时重新执行临时密钥交换，随后重新加入房间，并从本地偏移量之后请求持久化密文。

服务端是密钥包的不可信转发方。客户端签名验证和 TOFU 检查才是身份判断依据。旧客户端必须收到明确的升级错误，不允许继续使用明文消息、文件或语音路径。

## 私聊消息

私聊作用域由双方身份公钥按固定顺序组成，而不是由展示名组成。双方通过 X25519 和 HKDF 派生私聊根密钥，再分别派生消息密钥、文件包装密钥和语音密钥。持久化消息通过 `SEND_ENCRYPTED_MSG` 发送，并设置 `scope_type=dm`；服务端只能根据已注册的作用域参与者验证发送方和接收方，不得读取消息内容。

所有私聊消息都必须在离开客户端前加密。实时密文和同步密文使用同一条解密路径。缺少对端密钥或对端身份变化时，显示“[加密私聊 - 私聊密钥不可用]”，不得显示房间密码提示。该隔离同时解决 issue #5。

为支持客户端重启后的离线私聊解密，每条消息的内容密钥需要分别为双方包装。包装密钥由发送方的临时 X25519 密钥与每位参与者带签名的持久 X25519 预密钥派生。临时密钥之间的在线传输提供前向保密；持久接收方信封则有意提供离线投递能力。信封格式必须带版本，以便未来接入一次性预密钥双棘轮，而无需修改服务端存储表。

## 房间加密与认证

房间密码不得离开客户端。创建者生成随机房间 salt，使用 Argon2id 派生房间根密钥。服务端保存 salt 和由创建者生成的随机不透明房间访问令牌，但不保存密码验证值。访问令牌使用房间根密钥加密，并作为房间元数据分发。加入者在本地解密令牌，再通过 TLS WebSocket 将其作为授权凭证提交。该方案防止服务端获知房间密码，但需要明确承认：被入侵的服务端可以观察有效令牌并允许他人加入，不过仍无法派生房间密钥或解密内容。

房间根密钥分别派生消息密钥、文件包装密钥和语音密钥。已有房间消息迁移到统一 AEAD 信封。密码错误或密文被篡改只影响当前房间，不得影响私聊状态。

## 文件加密

每个文件生成随机 256 位文件密钥和随机文件标识。文件名、大小、摘要、分块大小和 MIME 类型等元数据必须加密。文件密钥使用当前房间或私聊的文件包装密钥包装。每个分块独立加密，nonce 由文件标识和分块索引生成，并通过附加认证数据绑定传输标识、索引、总分块数和发送方身份。

服务端中继、UDP 直传和 WebRTC DataChannel 使用相同的加密分块格式。接收端逐块验证完整性，并在最终重命名前根据加密元数据中的摘要再次校验。不得暴露部分写入的明文文件；传输失败时删除临时输出。服务端默认仍不持久化文件内容。

## 语音加密

每次通话根据私聊或房间语音密钥、新生成的通话标识和双方临时密钥包派生通话密钥。音频包包含发送方标识、通话标识、单调递增序列号和 ChaCha20-Poly1305 密文。中继 UDP 和直连 UDP 携带完全相同的加密数据包。接收端必须先完成认证再解码或播放，并拒绝重复或过旧的序列号。

对端身份未受信任、签名无效或协商协议缺少加密语音能力时，通话建立必须失败关闭，不允许回退到明文语音。

## 消息持久化与 TTL

服务端只保存带版本的密文信封和非秘密路由元数据。房间和私聊作用域分别拥有 `message_ttl_seconds` 和 `persist_messages`。面向用户的可选值严格限定为：

- 一天：86400 秒
- 一周：604800 秒
- 一个月：2592000 秒
- 一年：31536000 秒
- 永久：不设置过期时间

聊天窗口标题区域增加一个带工具提示的时钟图标按钮。菜单作用于当前显示的房间或私聊，并标记当前值。房间 TTL 仅允许房间创建者修改；私聊 TTL 由双方取更短的请求值，任意一方均可立即缩短保留期。服务端向所有在线作用域成员广播最终生效策略。客户端在时钟菜单中显示当前策略，更新后无需重新打开会话。

服务端计算 `expires_at`，同步查询时过滤过期记录，并由后台任务删除过期记录。“永久”使用空过期时间。客户端也必须忽略已经过期的密文。全局关闭持久化后，TTL 控件置为不可用，并说明消息仅实时中继。

## 错误处理

密码学错误必须结构化并区分作用域。签名无效、身份密钥变化、nonce 重放、信封格式错误、算法不支持和解密失败时，不得回退到明文。日志可以记录标识符和错误码，但不得记录明文、完整密文、密码、私钥、访问令牌或解密后的元数据。

界面必须区分房间密钥错误、私聊密钥错误、身份变化、协议不支持以及文件或语音密文损坏。一个作用域失败时，不得清除其他作用域的密钥或偏移量。

## 兼容与迁移

协议版本 3 改为严格模式，并要求具备认证密钥交换、加密消息、加密文件、加密语音、TTL 策略和离线同步能力。由于接受旧明文路径会违反 issue #7，旧客户端必须被拒绝并显示升级提示。已有加密房间历史可通过旧版信封解码器读取，但所有新发送内容必须使用新的认证信封。

现有客户端首次升级启动时生成身份密钥。现有服务端消息表原地迁移。不得导入或持久化历史明文记录。

## 测试与验收

单元测试覆盖身份持久化、指纹、签名密钥包、HKDF 上下文隔离、AEAD 篡改、nonce 唯一性、重放拒绝、房间密码派生、文件分块加密和语音包加密。

集成测试验证严格 HELLO 顺序、能力不匹配、TOFU 及身份变化、加密私聊实时及离线投递、房间同步、全部五档 TTL、TTL 权限、过期删除和关闭持久化。

端到端测试检查中继载荷和服务端存储，证明已知消息文本、文件字节、文件名和语音采样不会出现。GUI 测试覆盖时钟菜单、TTL 当前状态、身份变化警告和 issue #5 的私聊专用错误文案。最终提交前必须通过完整现有测试套件和打包冒烟测试。
